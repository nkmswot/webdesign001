How to Use Zustand for State Management in React

Zustand has become the go-to state management library for React in 2025 — simpler than Redux, more scalable than Context API, and zero boilerplate. Here's everything you need to get productive with it.

Why Zustand?

The state management landscape in 2025: Redux Toolkit is still solid for complex apps, but Zustand has won for most medium-complexity applications. It offers:

  • No providers, no wrapping — just import the hook
  • Zero boilerplate compared to Redux
  • No Context re-render problems (selective subscriptions)
  • Works outside React components (useful for testing)
  • Only 1KB gzipped

Basic Store

npm install zustand
// store/useCounterStore.ts
import { create } from 'zustand'

interface CounterStore {
  count: number
  increment: () => void
  decrement: () => void
  reset: () => void
}

export const useCounterStore = create((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
  reset: () => set({ count: 0 }),
}))
// Component — no Provider needed, just import
import { useCounterStore } from '@/store/useCounterStore'

export function Counter() {
  const { count, increment, decrement } = useCounterStore()

  )
}

Selectors — Avoid Unnecessary Re-renders

// Only re-renders when count changes, not when other store values change
const count = useCounterStore((state) => state.count)
const increment = useCounterStore((state) => state.increment)

// Anti-pattern: subscribes to entire store — re-renders on ANY change
const store = useCounterStore()  // ← avoid this for performance

Async Actions

// store/useUserStore.ts
interface UserStore {
  user: User | null
  loading: boolean
  error: string | null
  fetchUser: (id: string) => Promise
}

export const useUserStore = create((set) => ({
  user: null,
  loading: false,
  error: null,
  fetchUser: async (id) => {
    set({ loading: true, error: null })
    try {
      const res = await fetch(`/api/users/${id}`)
      const user = await res.json()
      set({ user, loading: false })
    } catch (e) {
      set({ error: e.message, loading: false })
    }
  },
}))

Persistence with localStorage

import { create } from 'zustand'
import { persist } from 'zustand/middleware'

interface ThemeStore {
  theme: 'light' | 'dark'
  setTheme: (theme: 'light' | 'dark') => void
}

export const useThemeStore = create()(
  persist(
    (set) => ({
      theme: 'light',
      setTheme: (theme) => set({ theme }),
    }),
    {
      name: 'theme-storage',  // localStorage key
      // partialize: (state) => ({ theme: state.theme })  // only persist specific fields
    }
  )
)
✅ Store Organization

Keep stores small and focused — one store per domain (useUserStore, useCartStore, useUIStore). Large monolithic stores defeat Zustand's selective re-render advantage.

FAQ

When should I use Zustand vs Context API?
Context API for values that change rarely (theme, locale, auth). Zustand for values that change frequently or need to be accessed by many components — Context causes re-renders across the entire tree on every change.
Can I use Zustand with Next.js Server Components?
No — Zustand is client-side only. For server-side state, use the URL, cookies, or database directly in Server Components. Add "use client" to any component that uses a Zustand store.
